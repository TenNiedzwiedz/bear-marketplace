# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project state

This is an early-stage **Laravel 13 / PHP 8.3** application named `bear-marketplace`. The frontend stack is **Inertia.js + Vue 3 + Vite with SSR** (a "monolith SPA" — no separate REST API for the app's own frontend; controllers return `Inertia::render(...)` and pass data as page props). It is otherwise a near-vanilla skeleton: the only routes are `/` (renders the `Welcome` Inertia page) and the `/up` health check, and the only domain model is `User`. There is no marketplace domain (listings, categories, company profiles, etc.) implemented yet — expect to be building it out. See `docs/zalozenia-projektowe.md` for the project's design assumptions (in Polish). `tests/` still contains only the framework's example tests.

## Commands

```bash
composer setup                 # First-time bootstrap: install deps, copy .env, key:generate, migrate, npm install + build
composer dev                   # Run everything at once (server + queue worker + `pail` log tailing + Vite) via concurrently
composer test                  # Clears config cache, then runs the full PHPUnit suite
php artisan test                       # Run tests directly
php artisan test --filter=SomeTest     # Run a single test class or method
php artisan test tests/Feature/Foo.php # Run one test file
./vendor/bin/pint              # Format code (Laravel Pint) — run before finishing changes
./vendor/bin/pint --dirty      # Format only uncommitted changes
php artisan tinker             # REPL for poking at models / the container
npm run dev                    # Vite dev server only (if not using `composer dev`)
npm run build                  # Build BOTH the client and SSR bundles (`vite build && vite build --ssr`)
php artisan inertia:start-ssr  # Run the Node SSR server (needs a built SSR bundle); required for SSR in production
```

## Architecture notes

- **Laravel 13 slim structure.** There is no `app/Http/Kernel.php` or `app/Console/Kernel.php`. Middleware, exception handling, and routing are configured in `bootstrap/app.php`; service providers are registered in `bootstrap/providers.php`. Register new middleware/exception behavior there, not in a Kernel. The `web` group appends `HandleInertiaRequests` (add data shared with every page in its `share()` method).
- **Inertia + Vue + SSR frontend.** Pages are Vue SFCs in `resources/js/pages/` (lowercase `pages` — matches `config/inertia.php` and matters on case-sensitive filesystems). A controller returns `Inertia::render('Welcome', [...props])` and the matching `resources/js/pages/Welcome.vue` receives them as props. Entry points: `resources/js/app.js` (client) and `resources/js/ssr.js` (SSR); the root Blade template is `resources/views/app.blade.php`. SSR is enabled by default (`INERTIA_SSR_ENABLED`); the SSR bundle builds to `bootstrap/ssr/` (gitignored) and is served by `php artisan inertia:start-ssr` on port 13714. Feature tests can assert against Inertia responses with `AssertableInertia`.
- **API requests already render JSON errors.** `bootstrap/app.php` sets exceptions to render as JSON for `api/*` paths, but there is no `routes/api.php` yet — add one via `->withRouting(api: ...)` (or `php artisan install:api`) when building API endpoints.
- **Database is MariaDB, not SQLite.** `DB_CONNECTION=mariadb` (see `.env.example`). You need a running MariaDB before migrating/testing locally against a real DB. The test suite (`phpunit.xml`) overrides to `DB_DATABASE=testing`.
- **Database-backed infrastructure.** Sessions, cache, and the queue all default to the `database` driver. Running the queue matters — `composer dev` starts a `queue:listen` worker for this reason. Cache/jobs/sessions tables come from the migrations in `database/migrations/`.
- **Local services via Sail.** `compose.yaml` defines the dev stack: the app (`laravel.test`), **MariaDB 11**, and **Mailpit** (SMTP on 1025, dashboard on 8025). Run with `./vendor/bin/sail up`.
- **Domain model.** Marketplace entities live in `app/Models/`: `User` (with `account_type`, plus moderation flags `is_admin` + `blocked_at`), `CompanyProfile` (1—1, only for company accounts), `Category` (self-referential tree via `parent_id`), `Listing` (belongs to a user + leaf category), `ListingImage`, `Report` (**polymorphic** — `reportable()` morphs to a `Listing` **or** a `User`; a short morph map `listing`/`user` is registered in `AppServiceProvider`). Statuses are **PHP enums in `app/Enums/`** (`AccountType`, `ListingStatus`, `ReportStatus`, `ReportReason`) cast on the models — treat them as the source of truth, and use `Listing::published()` for anything public-facing (it means status = active **and** the owner is not blocked). See `docs/zalozenia-projektowe.md` for the design rationale.
- **Seeders are development-only.** `DatabaseSeeder` bails out under `app()->isProduction()` and orchestrates one seeder per table (`CategorySeeder` → `UserSeeder` → `CompanyProfileSeeder` → `ListingSeeder` → `ListingImageSeeder` → `ReportSeeder`); they assume a fresh DB (`php artisan migrate:fresh --seed`) and generate fake data (accounts use password `password`). Do not rely on them in any non-dev environment.

## Conventions

- **Tests use PHPUnit 12, not Pest** — write `extends TestCase` class-based tests under `tests/Unit` or `tests/Feature` (the two configured suites). Note `laravel/pail` and the Pest plugin are present, but the suite is PHPUnit.
- **Frontend is Vue 3 (SFCs) + Tailwind CSS v4 + Vite 8.** Build UI as Inertia pages/components under `resources/js/`, not Blade views (`resources/views/` holds only the single `app.blade.php` shell). Tailwind is configured through the `@tailwindcss/vite` plugin (no `tailwind.config.js`); `vite.config.js` wires the `@vitejs/plugin-vue` plugin plus the client and SSR inputs. Fonts (Instrument Sans, Bricolage Grotesque, Space Mono) are self-hosted via the plugin's `bunny()` helper in `vite.config.js` — add weights there, don't link a CDN.
- **Design system.** Reusable UI lives in `resources/js/components/`: brand/atoms (`AppButton`, `SellerBadge`, `AppTag`, `SearchField`, `ListingCard`, `BearSeal`, `CategoryTile`, `CategoryGlyph`, `SectionBlock`, `EyebrowLabel`) and page chrome (`SiteHeader`, `SiteFooter`). Compose production views from these rather than restyling from scratch. Design tokens are defined in `resources/css/app.css`: a fixed brand palette in `@theme` (`--color-pine/paper/honey/moss/cranberry/taupe`) plus theme-aware semantic tokens (`--bg`, `--surface`, `--text`, `--accent`, `--seller-firma`, …) that swap under `prefers-color-scheme` and an explicit `data-theme` on `<html>`. Components read the semantic tokens via scoped styles — never hard-code hex. Note `--accent` (bright honey) is only for fills with dark ink on top; for honey-colored **text or icons on light grounds use `--accent-text`**, which stays legible. The live reference is the `/design-cheatsheet` page (`resources/js/pages/DesignCheatsheet.vue`); `/` is served by `HomeController` (categories with subtree counts, latest listings, stats) into `Home.vue`, built entirely from these components. Controllers act as the view-model — they shape DB rows into the exact props a page expects, keeping presentation logic thin in Vue. Shared shaping lives in `app/Support/` (e.g. `ListingCardPresenter::present()` turns a `Listing` into `ListingCard` props — price, seller badge, image URL — and is reused by `HomeController` and `ListingController`); reuse it for any new view that renders listing cards. The listings index (`/ogloszenia` → `ListingController@index`, `resources/js/pages/Listings/Index.vue`) shows the filter pattern: validate query params, build the query with `->when()`, `paginate()->withQueryString()->through(presenter)`, and drive it from the front with `router.get(url, params, { preserveState, preserveScroll, replace })`. The detail page (`ListingController@show`, `/ogloszenia/{listing}`, `Listings/Show.vue`) `abort_unless`s the listing is active — only active listings are public; its seller panel links through to the seller's public profile. That profile (`SellerController@show`, `/sprzedawca/{user}`, `resources/js/pages/Sellers/Show.vue`) is public and shows a header (initials avatar, seller badge, active-listing count, member-since, company facts) plus the seller's `published()` listings as a paginated `ListingCard` grid (reusing `ListingCardPresenter`), `EmptyState` when none. `ListingCard` takes an `href` and becomes a stretched-link card. The add/edit form (`ListingController@create/store/edit/update`, `/wystaw` + `/ogloszenia/{listing}/edytuj`, shared `Listings/Form.vue`) uses Inertia `useForm` with `forceFormData` for image upload (updates spoof `_method: put`); images land on the `public` disk (run `php artisan storage:link`), a publish/draft action sets the status, and the owner is the authenticated user (edit/update authorize ownership via `ListingPolicy`). Forms use the reusable `FormField` (label/error/hint) wrapper. The panel listings table also drives lifecycle actions — `ListingController@end`/`reactivate` (status transitions, guarded against invalid ones) and `@destroy` (deletes the listing and its uploaded image files) — each authorized via `ListingPolicy`, called from the front with `router.patch`/`router.delete`. The user panel is split into sections that share `PanelLayout` (site header + user sidebar with section nav + slot): `DashboardController@overview/listings/messages/account` under the `/panel` route group render `resources/js/pages/panel/{Overview,Listings,Messages,Account}.vue`. It is adaptive (private vs company) and runs on the authenticated user. Listing statuses render via `StatusPill` + `resources/js/lib/listingStatus.js` (the shared status→label→color map, backed by the semantic `--status-*` tokens in `app.css`). `EmptyState` is the shared branded empty screen.
- **Auth.** Session-based, custom-built on the design system (no Breeze): guest-only `/rejestracja` + `/logowanie` (`RegisteredUserController`, `AuthenticatedSessionController`, `Auth/{Register,Login}.vue` in an `AuthLayout`), `/wyloguj` behind `auth`. Registration captures the account type and, for companies, creates the `CompanyProfile`. The panel and listing create/edit routes sit behind the `auth` middleware group in `routes/web.php` (`/logowanie` is the named `login` redirect target); `ListingPolicy` (auto-discovered) guards edit/update ownership via `Gate::authorize`. `HandleInertiaRequests` shares `auth.user`, which `SiteHeader` reads to switch between the logged-out and logged-in controls.
- **Moderation & reporting.** Any signed-in user can report a listing or an account: the reusable `ReportDialog` component (a modal, reason + note, mirrors `App\Enums\ReportReason` via `resources/js/lib/reportMeta.js`) POSTs to `/zglos` → `ReportController@store`, which rejects self-reports and de-dupes pending ones. It's wired into `Listings/Show.vue` (listing + seller) and `Sellers/Show.vue`, shown only to non-owners (`auth.user.id` is shared for that check). **Admins** (`User::isAdmin()`, `is_admin` flag; shared as `auth.user.isAdmin` and surfaced as a header link) reach the panel at `/admin`, guarded by the `admin` middleware alias (`EnsureUserIsAdmin`, registered in `bootstrap/app.php`). `Admin\DashboardController` renders `resources/js/pages/admin/{Overview,Reports,Listings,Users}.vue` inside the shared `AdminLayout` (staff-toned, cranberry accent, pending-reports badge); `Admin\ModerationController` holds the actions — hide/restore/delete a listing, block/unblock a user (an admin can't block themselves or another admin), resolve/dismiss a report. **Blocking**: `blocked_at` is deliberately *not* mass-assignable (set directly in the controller); a blocked account is rejected at login and signed out mid-session by `EnsureUserIsNotBlocked` (appended to the `web` group), and its listings drop out of `published()`. Deleting a listing uses `Listing::deleteWithImages()` (removes uploaded files); admin deletion also clears the listing's polymorphic reports first (no DB cascade on morphs). Report status renders via `ReportStatusPill` + the shared `ReportRow` component.
- **Flash toasts.** `HandleInertiaRequests` also shares `flash.success` / `flash.error` (from session). To raise a toast, redirect with `->with('success', '…')` (or `'error'`). The global `Toaster` component — mounted once at the app root alongside the page in `resources/js/app.js` and `ssr.js` — reads the flash on load and on every Inertia `router.on('success')` and auto-dismisses. It is client-only (renders an empty container during SSR).
- Code style is enforced by **Pint** (Laravel preset). Match the existing formatting; run Pint rather than hand-formatting.
